# Design Tokens with Tailwind v4 and shadcn/ui

Tailwind CSS v4 replaced `tailwind.config.ts` with a CSS-first configuration model. Tokens live in `@theme` blocks inside your stylesheet. This template uses that model alongside shadcn/ui's semantic token conventions to produce a fully themeable design system with zero JavaScript configuration.

## The old way vs the new way

In Tailwind v3 you extended the theme in a JavaScript config file:

```ts
// tailwind.config.ts (v3)
export default {
  theme: {
    extend: {
      colors: {
        primary: 'hsl(220 90% 56%)',
        background: 'hsl(0 0% 100%)',
      },
    },
  },
}
```

In Tailwind v4 you declare the same tokens in CSS using `@theme`:

```css
/* globals.css (v4) */
@import "tailwindcss";

@theme {
  --color-primary: oklch(55% 0.22 264);
  --color-primary-foreground: oklch(98% 0.01 264);
  --color-background: oklch(100% 0 0);
  --color-foreground: oklch(14.5% 0.025 264);
}
```

Every `--color-*` variable you declare in `@theme` automatically becomes a Tailwind utility: `bg-primary`, `text-foreground`, `border-primary-foreground`.

## Semantic tokens

shadcn/ui's convention uses _semantic_ token names — names that describe purpose, not appearance:

| Token | Utility | Meaning |
|---|---|---|
| `--color-background` | `bg-background` | Page background |
| `--color-foreground` | `text-foreground` | Body text |
| `--color-muted` | `bg-muted` | Subtle fills |
| `--color-muted-foreground` | `text-muted-foreground` | Subdued text |
| `--color-destructive` | `bg-destructive` | Error / danger |
| `--color-border` | `border-border` | All borders |
| `--color-ring` | `ring-ring` | Focus ring |

Using semantic names means dark mode is a single override block — you change what the tokens resolve to, not every class across the codebase.

## Dark mode

Tailwind v4 uses `@custom-variant` for class-based dark mode:

```css
@custom-variant dark (&:where(.dark, .dark *));

.dark {
  --color-background: oklch(14.5% 0.025 264);
  --color-foreground: oklch(98% 0.01 264);
  --color-muted: oklch(22% 0.02 264);
  --color-muted-foreground: oklch(65% 0.02 264);
}
```

Toggle the `dark` class on `<html>` and every token flips. No JavaScript-driven class juggling per component.

## OKLCH for color

This template uses OKLCH instead of HSL. OKLCH (Oklch) is a perceptual color space where equal numeric steps produce visually equal differences in lightness and chroma — HSL does not guarantee this. Practically this means:

```css
/* These three are perceptually equidistant in OKLCH */
oklch(40% 0.18 264)
oklch(55% 0.18 264)
oklch(70% 0.18 264)

/* HSL "equivalents" vary in apparent brightness */
hsl(220 50% 35%)
hsl(220 50% 50%)
hsl(220 50% 65%)
```

Use the [OKLCH color picker](https://oklch.com/) to sample values when extending the palette.

## Component tokens

For component-specific tokens, follow the same pattern at a narrower scope:

```css
@theme {
  --color-card: oklch(100% 0 0);
  --color-card-foreground: oklch(14.5% 0.025 264);

  --radius-sm: 0.25rem;
  --radius-md: 0.375rem;
  --radius-lg: 0.5rem;
}
```

Then use them in components:

```tsx
<div className="rounded-lg bg-card text-card-foreground border border-border p-6">
  Card content
</div>
```

The result is a component whose appearance is entirely controlled by token overrides — swap the tokens for a different brand in seconds.
